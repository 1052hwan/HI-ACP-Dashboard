# [운영 보고서] 서버리스 수문 데이터 자동 수집 시스템 구축 결과

본 보고서는 로컬 PC 가동 없이 클라우드 환경에서 24시간 작동하는 수문 데이터 자동화 워크플로우의 구조와 운영 방법을 상세히 설명합니다.

---

## 1. 시스템 워크플로우 개요 (Workflow Overview)

전체 시스템은 Gmail(수신) → Google Apps Script(처리) → Supabase(저장) → GitHub Pages(시각화)의 4단계 클라우드 체인으로 구성됩니다.

### Step 1: Gmail 감시 및 파일 식별
* 작동 방식: 구글 앱스 스크립트(GAS)가 설정된 시간(1시간마다)에 따라 지메일함을 스캔합니다.
* 식별 기준: 
    - 제목에 _rawdata 키워드가 포함된 메일
    - 첨부파일이 있는 메일
* 중복 방지: 처리가 완료된 메일은 자동으로 '별표(Star)' 처리를 하여 다음 실행 시 중복 수집되지 않도록 설계되었습니다.

### Step 2: 클라우드 내 인메모리(In-memory) 데이터 처리
* 압축 해제: 수신된 ZIP 파일을 서버 메모리상에서 즉시 압축 해제합니다.
* 데이터 파싱: 파일 확장자(.adv, .esv)에 따라 서로 다른 데이터 구조를 분석합니다.
* 논리적 월(Logical Month) 계산: 
    - 기준: 당월 1일 00:10 ~ 익월 1일 00:00
    - 예: 2026-05-01 00:00 데이터는 자동으로 4월 데이터로 분류되어 저장됩니다.

### Step 3: Supabase 데이터베이스 적재
* 실시간 전송: 정제된 데이터를 Supabase REST API를 통해 직접 전송합니다.
* 중복 업데이트(Upsert): 동일 지점, 동일 시간에 이미 데이터가 존재할 경우 에러를 내지 않고 최신 값으로 업데이트합니다.
* 로그 기록: 매 실행마다 collection_logs 테이블에 파일 수, 유효 레코드 수, 결측치 리스트를 기록합니다.

### Step 4: 대시보드 시각화 (GitHub Pages)
* 배포 환경: GitHub Pages를 통해 전 세계 어디서든 웹브라우저로 접속 가능합니다.
* 동적 로딩: 대시보드 실행 시 sitename.txt 파일을 읽어 관측소 코드를 한글 명칭으로 자동 변환합니다.
* 실시간성: DB에 데이터가 들어오는 즉시 대시보드의 숫자가 갱신됩니다.

---

## 2. 운영 및 관리 가이드

### 2.1 관측소 추가 방법
* 새로운 관측소가 추가되면 sitename.txt 파일에 [코드, 이름] 형식을 한 줄 추가하여 GitHub에 업로드(Commit)하면 즉시 대시보드에 반영됩니다.

### 2.2 수집 상태 모니터링
* 정상 처리: 지메일함에서 메일 옆에 노란색 별표가 붙어 있다면 성공적으로 수집된 것입니다.
* 오류 확인: 만약 메일이 왔음에도 별표가 붙지 않는다면, 구글 앱스 스크립트의 '실행 로그'를 통해 오류 원인을 파악할 수 있습니다.

### 2.3 데이터 수정 및 재수집
* 잘못된 데이터가 들어와서 다시 수집해야 할 경우, 지메일에서 해당 메일의 별표를 해제하고 앱스 스크립트를 수동으로 [실행]하면 다시 처리되어 DB가 업데이트됩니다.

---

## 3. 시스템의 장점
1. 무결성: 사람이 개입하지 않아 데이터 누락 및 전산 오입력이 원천 차단됩니다.
2. 비용 제로: 구글과 GitHub의 무료 티어 내에서 완벽하게 구동되어 유지비용이 발생하지 않습니다.
3. 확장성: Supabase DB를 기반으로 하므로 향후 AI 분석 모델이나 더 복잡한 통계 시스템으로의 확장이 용이합니다.

이상으로 수문 데이터 자동 수집 시스템의 구축 및 운영 보고를 마칩니다.
